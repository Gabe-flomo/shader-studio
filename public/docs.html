Shader Studio

A visual node editor for writing GPU shaders — without writing GPU shaders. Connect nodes, see pixels move, understand why.

There's something deeply human about the desire to make things glow. To coax color out of mathematics. To look at a blank screen and think: I could put something beautiful there.

Shader Studio is built for that impulse. It takes the arcane craft of GPU programming — a discipline historically gated behind unfriendly syntax and counter-intuitive mental models — and asks: what if you could just... see it?

Every node you connect is a line of GLSL being written for you. Every parameter you drag is a uniform being patched into a running program. The GPU is doing what it has always done — running the same function on millions of pixels simultaneously — but now you're the one deciding what that function looks like, without needing to speak its native tongue.

What is a Shader?

Understanding the mental model that makes everything else click.

Consider the modern screen. One million, two hundred and eighty thousand pixels, give or take, each one a small burning question: what color should I be?

A shader is the answer. More precisely, a fragment shader is a program that runs once per pixel, in parallel, on thousands of GPU cores simultaneously. It receives a coordinate — where on screen this particular pixel lives — and must return a color. That's it. That's the entire contract.

// A fragment shader, in its simplest possible form:
// Given this coordinate... what color am I?
vec3 color = someComputationOf(coordinate);
gl_FragColor = vec4(color, 1.0);

The profundity — and the source of all the visual complexity you see in shader art — is that the same program runs on every pixel. There is no loop over pixels. There is no "draw this shape here." There is only: given my position, what should I be?

CPU vs GPU thinking

CPU programming is sequential. You do step 1, then step 2, then step 3. You iterate over things. You check conditions. You maintain state between operations.

GPU programming is massively parallel. You write one function. The GPU calls it simultaneously for every pixel. There is no "previous pixel." There is no shared state between them. Each pixel is an island, doing its own computation, unaware of its neighbors.

This is why shaders produce the effects they do. A circle isn't drawn by tracing around its perimeter — it's produced by every pixel asking "am I within distance r of the center?" and coloring itself accordingly. A noise texture isn't painted stroke by stroke — it's every pixel evaluating the same mathematical function at its own coordinate and getting a slightly different answer.

What Shader Studio does

Shader Studio compiles your node graph into a real GLSL fragment shader and runs it on your GPU in real time. Every node represents a piece of GLSL code. Every connection is a variable being passed between operations. When you change a parameter, Shader Studio patches the corresponding uniform and the GPU re-runs the shader immediately — every pixel, simultaneously, in the time it takes you to release the mouse button.

The GLSL panel (toggle with ⌘\) shows the exact shader being compiled. Click any node to highlight the code it contributes. This is the single most useful tool for understanding how your graph maps to the GPU.

UV Space

The coordinate system that everything in Shader Studio flows from.

There's something profound about the question of where you are. A pixel could describe its position as raw screen coordinates — "I am pixel 847, 312" — but that answer is contingent, fragile, hardware-specific. It means something entirely different on a 1080p monitor than on a 4K display. The shape you build for one resolution becomes the wrong shape on another.

UV space is the resolution to this existential problem. It describes position not in pixels, but in proportions. And in Shader Studio specifically, it describes position relative to the center of the screen, in a way that accounts for aspect ratio so that circles look like circles.

The UV node

When you place a UV node, it outputs a vec2 with these properties:

  • Origin at center — (0, 0) is the middle of the canvas.
  • Range: roughly -1 to +1 in both axes, with the exact horizontal range adjusted for aspect ratio.
  • Aspect-corrected — on a widescreen canvas, X extends beyond ±1, but Y stays within ±1. This means a circle drawn at radius 0.5 looks perfectly round regardless of canvas dimensions.
// What the UV node actually computes:
vec2 uv = (vUv - 0.5) * 2.0;
uv.x *= resolution.x / resolution.y;

This is why the UV node is almost always the first thing in any graph. It's your foundation — the coordinate system everything else builds on.

Using UV as input

Most nodes that work in 2D space expect a vec2 input labeled "UV" or "P". They use this to decide where in space to evaluate whatever they compute. Feed them the UV node's output and they'll evaluate relative to screen center.

UV transforms propagate downstream

One of the most important things to internalize: in Shader Studio, you don't move shapes. You move the coordinate space, and then evaluate shapes in the transformed space. When you add a Rotate 2D or Fract node, you're transforming UV before it reaches the SDF. The SDF stays at the origin — you moved the floor under it.

Your First Shader

From blank canvas to glowing circle in five steps.

Witness, for a moment, the blank canvas. A rectangle of pure darkness. No color. No form. Just the quiet of a GPU awaiting instruction. In the time it takes to add five nodes, we will give it something worth looking at.

  1. Add a UV node Press A to open the node palette and search "UV", or press U as a shortcut. This outputs the screen coordinate as a centered, aspect-corrected vec2. It goes in the output, so you'll see a gradient — red on the right, dark on the left.
  2. Add an Output node Press O. Drag a connection from the UV node's output to the Output node's Color input. The canvas shows a gradient — you're looking at raw UV coordinates being interpreted as color. This is normal and useful.
  3. Add a Circle SDF node Open the palette, search "circle". Connect UV → Circle SDF's P input. The SDF outputs a float — the signed distance from each pixel to the circle edge. Connect it to Output. You'll see a gray gradient that's lighter in the center — that's the raw distance field.
  4. Add a Make Light node Search "make light". Connect Circle SDF's Distance → Make Light's Distance input. Make Light computes brightness / abs(distance) — a glow that's bright near zero and falls off with distance. Connect Make Light's Glow → Output. You'll see a white glow ring. Adjust the Brightness slider to taste.
  5. Add a Palette Preset node Search "palette preset". Add a Time node (T). Connect Time → Palette's T input. Connect Palette's Color → a Scale Color node, Scale Color → Output. Now connect Make Light's Glow → Scale Color's Scale. You have an animated, color-cycling glowing circle.

Once you have this working, try adding a Tone Map node before Output (ACES mode). It handles the HDR brightness gracefully and gives the glow a more cinematic quality.

What just happened

You defined a function: for every pixel, compute its distance to a circle centered at the origin, turn that distance into a brightness value, color it with a cosine palette driven by time, and output the result. That function ran on every pixel simultaneously, 60 times per second. The animation is just time flowing into the palette's input — the GPU recomputes the whole thing each frame.

The Interface

Everything on screen and what it does.

The node canvas

The main area where you build your graph. Scroll to zoom, click and drag on empty space to pan. Nodes can be moved by dragging their header.

  • Right-click empty space — opens the node palette
  • Press A — opens the searchable node palette
  • Press F — fit all nodes in view
  • Click a node — select it; its parameters appear in the right panel
  • Drag output → input — create a connection. Colors indicate type compatibility.
  • Click a connection — delete it

Node anatomy

Each node has a header with its name, input sockets on the left, and output sockets on the right. Socket colors map to data types: float yellow, vec2 teal, vec3 green, vec4 purple.

Most nodes have parameter sliders that appear when selected. If a socket is wired, the connected value overrides the slider. If unwired, the slider value is used as a constant.

The preview canvas

The live preview renders your shader in real time. It updates every frame when animated, or immediately when you change a parameter. Resize it by dragging the divider. The preview respects aspect ratio — if you make it wider, the UV node's X range expands.

The GLSL panel

Toggle with ⌘\. Shows the compiled fragment shader exactly as it runs on the GPU. Click any node in the canvas — the corresponding lines in the GLSL panel highlight. This is the most direct way to understand what your graph compiles to.

The parameter panel

Right sidebar. Shows all parameters for the selected node. Sliders, dropdowns, color pickers, and vec2/vec3 inputs depending on the node type. Any slider can be overridden by wiring a float output to the corresponding input socket.

The minimap

Bottom-right of the canvas. Shows a bird's-eye view of your entire graph. Click or drag to navigate. Useful when graphs get large.

Node inline preview

Small nodes can show a tiny live preview of their output directly on the canvas. Toggle this per-node. Useful for verifying what a noise or SDF node is producing without wiring it to output.

Data Types

The language of values flowing between nodes.

In GLSL — and by extension in Shader Studio — every value has a type. A type tells you how many numbers are bundled together and how to interpret them. Most of what you'll work with falls into four families:

TypeWhat it holdsTypical use
floatA single decimal numberDistance, brightness, time, opacity, any scalar
vec2Two floats: (x, y)UV coordinates, 2D positions, offsets
vec3Three floats: (x, y, z) or (r, g, b)Colors, 3D positions, normals
vec4Four floats: (r, g, b, a)Colors with alpha, homogeneous coordinates

Why this matters

Socket colors in Shader Studio directly reflect these types. You can only connect an output to an input of the same type (or compatible types). When you try to connect a float to a vec2 input, the connection won't be accepted — the compiler can't know which two floats you meant to pair.

Swizzling

GLSL lets you extract and rearrange components with dot notation. This is called swizzling, and it's worth knowing because some nodes expose it:

vec3 color = vec3(0.2, 0.8, 0.5);
float r = color.r;        // 0.2  (same as color.x)
float g = color.g;        // 0.8  (same as color.y)
vec2 rg = color.rg;       // vec2(0.2, 0.8)
vec3 flip = color.bgr;    // vec3(0.5, 0.8, 0.2) — reversed channels

The spatial aliases .x .y .z .w and color aliases .r .g .b .a are interchangeable — uv.x and uv.r refer to the same component.

Numbers in GLSL

GLSL is strict about floats. 1 is an integer literal. 1.0 is a float. Mixing them without explicit casting causes compiler errors. Shader Studio handles this for you in node outputs, but if you're writing expressions in the Expr node, always use decimal points: 0.5 not 0, 2.0 not 2.

vec3 doubles as both "color" and "3D position" depending on context. This is fine — they're the same underlying type. A vec3 of (1.0, 0.0, 0.0) means "red" in a color context and "point on the X axis" in a geometry context.

Sources

Nodes with no inputs — the raw material that everything else is built from.

Source nodes are where data enters the graph. They have no input sockets — they generate values from the universe outside the shader (screen coordinates, elapsed time, user input, uploaded files) and offer them as outputs you can wire into anything downstream.

UV → vec2

The centered, aspect-corrected screen coordinate. Origin at center, range approximately ±1 vertically. The X range extends beyond ±1 on wide canvases. This is your primary coordinate input for almost everything. See UV Space for a full explanation.

Pixel UV → vec2

Raw pixel UV: gl_FragCoord.xy / resolution.y. Origin is bottom-left. X extends to the aspect ratio. Use this for shaders that need to work in pixel-proportional space rather than centered space.

Time → float

Elapsed time in seconds since the shader started. Wire this into any parameter to animate it. The most common pattern: connect Time into a Sin node to get a smooth oscillating value, or directly into palette/noise nodes that accept a time input. Press T to place one instantly.

Mouse → vec2, float, float

The current cursor position in the same centered UV space as the UV node. Outputs three values: the full UV position, the X component as a float, and the Y component as a float. Wire this into an SDF's offset input to make shapes follow your cursor.

Loop Index → float

When placed inside an iterated group, this outputs the current iteration number as a float (0.0, 1.0, 2.0...). Outside an iterated group it always outputs 0.0. This is the key to making each iteration of a loop do something slightly different. See Iterated Groups.

Constant → float

A fixed float value with a slider. Wire a float input to override the slider dynamically. Useful for parameterizing values you want to control from outside a group, or for wiring a specific literal value into a node that expects a socket connection. Press Shift+F to place one.

Prev Frame → vec3, float

Samples the previous frame's rendered output at a given UV coordinate. Enables feedback loops — the shader reads its own prior output. Use this for paint trails (blend current with 95% of previous), reaction-diffusion patterns, and fluid simulations. Outputs color (vec3) and alpha (float).

Prev Frame makes the shader stateful — it needs a framebuffer to store the previous frame. The preview rendering mode changes when this node is present. Stateful shaders cannot be exported as standalone fragments without a rendering harness.

Texture Input → vec3, float

Samples an uploaded image at a given UV coordinate. Click the node to upload an image. Connect a UV (or transformed UV) to control where on the image is sampled. Outputs RGB color and alpha separately.

Audio Input → float per band

Load an audio file and get per-frequency-band amplitude as floats between 0 and 1. Add multiple bands to react to bass, mids, and highs independently. Use Full Spectrum mode for a single overall volume value. Wire amplitude into SDF scale, palette phase, or any float input to create audio-reactive visuals.

Transforms

You don't move the shape. You move the space the shape lives in.

Consider what it means to "rotate a circle." In a traditional drawing program, you'd rotate the circle's vertices. In a shader, there are no vertices. There is only a question — am I inside this circle? — asked at every pixel.

To rotate the circle, you rotate the coordinate before asking the question. Instead of asking "is this pixel inside the circle?", you ask "is this pixel, rotated backward by the rotation amount, inside the circle?" The circle appears rotated. The math is the same.

This is the foundational insight behind all UV transforms: they modify the coordinate that gets fed into downstream nodes. Everything downstream sees a different version of space.

Fract / Tile → vec2

Tiles space by applying fract(uv * scale) - 0.5. Every tile is a copy of the same centered coordinate space. Feed a tiled UV into an SDF and you get infinite repetitions of that shape. The Scale parameter controls how many tiles appear across the screen.

// What Fract does:
vec2 output = fract(input * scale) - 0.5;

Rotate 2D → vec2

Rotates a vec2 by an angle in radians. Connect a Time node (multiplied by a speed) into the Angle input to spin things. The rotation is around the origin (0,0) — if you want to rotate around a different point, translate first, rotate, then translate back using a Displace node.

Displace → vec2

The "bring your own warp field" node. Takes any vec2 as an offset and adds it to the input UV, scaled by an Amount parameter. Plug in noise, a math expression, or any other vec2 source to use it as a displacement field. This is the most flexible warp — the other warp nodes are specialized versions of this idea.

UV Warp (Jitter) → vec2

Hash-grid jitter — intentionally blocky, pixelated displacement. Good for glitch effects, mosaic distortion, and digital noise aesthetics. The grid structure is visible by design.

UV Warp (Smooth) → vec2

Bilinear value-noise warp with no visible grid edges. Flowing, organic displacement. Parameters: Strength (how far pixels move), Scale (frequency of the warp pattern), Speed (animation rate). The most generally useful warp for natural-looking distortion.

UV Warp (Curl) → vec2

Divergence-free curl noise. Particles in this field neither converge nor diverge — they flow in closed loops, giving a fluid or smoke-like motion. The mathematical guarantee of divergence-freedom is what separates curl warp from smooth warp aesthetically.

UV Warp (Swirl) → vec2

Rotational twist that's strongest at the center and decays outward. Twist sets the maximum rotation angle. Falloff controls how quickly the rotation fades with distance. Animates smoothly with time. Center X/Y lets you shift the vortex origin.

opRepeat → vec2

Infinite domain repetition: mod(p + 0.5*s, s) - 0.5*s. Tiles space every s units. Unlike Fract, opRepeat doesn't center each tile — it tiles in a way that's mathematically consistent with SDF operations, making it the right choice when you need to tile an SDF and then perform boolean operations on the result.

opRepeatPolar → vec2

Polar domain repetition — creates N-fold rotational symmetry. Feed it a UV and the number of segments, and it folds space so that evaluating any SDF inside it produces N copies arranged radially. The building block for mandalas, radial patterns, and snowflake-like geometry.

// Classic mandala pipeline:
// UV → opRepeatPolar(6) → Rotate2D → sdBox → Make Light → Palette → Output

Vectors, Matrices & Transformations

The mathematical objects that make shader geometry possible.

There's something deeply satisfying about the moment a coordinate stops being a pair of numbers and starts being a thing you can rotate, scale, and project. This is the domain of linear algebra — and while you don't need to prove theorems to use Shader Studio, understanding the objects you're working with unlocks everything from smooth rotations to full 3D cameras.

Vectors

A vector is an ordered list of numbers. In GLSL, vec2, vec3, and vec4 are vectors of 2, 3, and 4 components respectively. You can think of them two ways depending on context:

  • As a point — a location in space. vec2(0.5, 0.3) is the point 0.5 units right, 0.3 units up from the origin.
  • As a direction + magnitude — an arrow. vec2(1.0, 0.0) points right with length 1. vec2(0.0, 2.0) points up with length 2.

The same math works on both interpretations. Whether you call it "a position" or "a displacement" depends on what you're doing with it.

The dot product, practically

The dot product is one of the most useful operations in shader code. Given two normalized vectors, dot(a, b) returns the cosine of the angle between them. This single fact powers most lighting calculations:

vec3 normal = ...; // surface normal, pointing outward
vec3 lightDir = normalize(vec3(1.0, 1.0, 0.5));

// How much does this surface face the light?
float diffuse = max(dot(normal, lightDir), 0.0);
// At 0° (facing directly): dot = 1.0 → full brightness
// At 90° (side-on):        dot = 0.0 → no contribution
// Below 90° (facing away): dot < 0   → clamped to 0

Matrices

A matrix is a grid of numbers that transforms vectors when you multiply them together. In GLSL: mat2 is a 2×2 matrix, mat3 is 3×3, mat4 is 4×4. You multiply a matrix by a vector to apply the transformation the matrix encodes.

// A 2D rotation matrix for angle θ:
mat2 rotation = mat2(
    cos(theta), -sin(theta),  // column 0
    sin(theta),  cos(theta)   // column 1
);

vec2 rotatedUV = rotation * uv;

GLSL matrices are column-major: the constructor fills columns, not rows. mat2(a, b, c, d) creates the matrix with column 0 = (a, b) and column 1 = (c, d).

The four fundamental 2D transformations as matrices

TransformMatrixNotes
Rotation by θ mat2(cos θ, sin θ, -sin θ, cos θ) Rotates counterclockwise. The Rotate 2D node does this.
Scale by (sx, sy) mat2(sx, 0, 0, sy) Stretches independently per axis.
Uniform scale by s mat2(s, 0, 0, s) Same as multiplying the vec2 by a scalar: uv * s.
Shear (skew) mat2(1, 0, k, 1) Slides the Y axis. Used in parallelogram / kaleidoscope patterns.
Translation cannot be represented by a 2×2 matrix — it's not a linear transformation. To translate, just add a vec2: uv + offset. For 3D, mat4 uses a trick called homogeneous coordinates to encode translation in the 4th column, which is why 3D camera matrices are 4×4.

You can chain transformations by multiplying matrices together, or equivalently in Shader Studio, by connecting transform nodes in sequence. The order matters — matrix multiplication is not commutative:

Polar coordinates — a non-matrix transform

Not all useful coordinate transforms are linear. Polar coordinates convert (x, y) into (radius, angle):

vec2 uv = ...; // Cartesian: (x, y)
float r     = length(uv);         // distance from origin
float angle = atan(uv.y, uv.x);   // angle in radians [-π, π]
vec2 polar  = vec2(r, angle);

Working in polar space is how you build radially symmetric patterns. Repeat the angle with mod(angle, 2π/N) to get N-fold symmetry. Animate the angle with time to get rotation. Use r directly as an SDF input to get ring patterns.

The 3D camera matrix

In 3D ray marching, a camera matrix (often called a "look-at" matrix) transforms a ray direction from screen space into world space. Shader Studio's camera nodes build this for you, but the math is:

// Look-at camera: builds an orthonormal basis
// ro = camera position, ta = target point, cr = roll angle
mat3 setCamera(vec3 ro, vec3 ta, float cr) {
    vec3 cw = normalize(ta - ro);          // forward
    vec3 cp = vec3(sin(cr), cos(cr), 0.0); // up hint
    vec3 cu = normalize(cross(cw, cp));    // right
    vec3 cv = normalize(cross(cu, cw));    // up (corrected)
    return mat3(cu, cv, cw);
}

// Usage per pixel:
vec2 p = (fragCoord - 0.5 * resolution) / resolution.y;
vec3 rayDir = normalize(cameraMatrix * vec3(p, focalLength));

The result is a ray that starts at the camera position and points toward whatever that pixel "looks at" in the scene. The SDF scene is then sampled along this ray.

2D Signed Distance Functions

The geometry system at the heart of Shader Studio's 2D visuals.

A signed distance function answers a single question: how far is this point from the nearest surface? The sign tells you which side you're on — negative means inside, positive means outside, and zero means exactly on the edge.

This is profound. A circle is no longer "a set of points equidistant from a center." It is the zero level set of the function length(p) - r. Every pixel evaluates this function. Those that get back a value near zero are near the edge. Those that get back a large positive value are far outside. Those that get back a negative value are inside.

float sdCircle(vec2 p, float r) {
    return length(p) - r;
}
// p = pixel coordinate, r = radius
// Returns: negative inside, 0 on edge, positive outside

From distance to color

The raw distance value is just a float. To make it visible, you need to decide what to do with it:

float d = sdCircle(uv, 0.4);

// Hard fill — sharp edge:
float fill = step(0.0, -d);           // 1.0 inside, 0.0 outside

// Smooth edge — anti-aliased:
float smooth = smoothstep(0.01, -0.01, d);  // soft edge 2px wide

// Glow — bright near zero, fades with distance:
float glow = 0.02 / abs(d);           // the classic shader glow

// Outline only — ring on the surface:
float ring = smoothstep(0.02, 0.0, abs(d));

The Make Light node

The Make Light node encodes the glow formula: brightness / abs(distance). It's the most common way to turn an SDF into something that looks good. Use the Brightness parameter to control intensity, and wire the output into a Scale Color node (to tint it) before Output.

Available 2D SDF nodes

Shader Studio includes the full IQ SDF library. The Shape SDF node lets you pick from all of them in a single node via a dropdown:

  • Circle — length(p) - r. The simplest and most used.
  • Box / Rounded Box — Axis-aligned rectangle, optionally rounded.
  • Segment — Distance to a line segment between two points A and B.
  • Ellipse — Stretched circle with independent radii.
  • Triangle / Isosceles Triangle — Equilateral or custom-proportioned triangle.
  • Pentagon, Hexagon, Octagon — Regular polygons.
  • Star (5-point and N-point) — Stars with configurable inner/outer radius ratio.
  • Arc / Ring / Pie — Circular slice and ring variants.
  • Cross / Rounded X — Plus sign and rounded cross shapes.
  • Parabola, Hyperbola — Conic section distance fields.
  • Moon, Vesica — Lens-shaped geometry.
  • Horseshoe, Tunnel, Stairs — Architectural/decorative shapes.

Boolean operations

Because SDFs are just floats, you can combine them with simple math:

float d1 = sdCircle(uv, 0.4);
float d2 = sdBox(uv, vec2(0.3, 0.2));

float union_      = min(d1, d2);       // both shapes
float subtract    = max(-d1, d2);      // d2 minus d1
float intersect   = max(d1, d2);       // only the overlap
float smooth_union = smin(d1, d2, 0.1); // blended/melted union

Smooth union (smin) is what gives shader geometry that organic, melted-together look. The third parameter controls the blend radius.

opRepeat and opRepeatPolar

opRepeat tiles space every S units — apply it to a UV before an SDF to get infinite tiling. opRepeatPolar creates N-fold rotational symmetry. These are in the Spaces category and are the correct building blocks for grid and radial layouts of SDFs.

Rounding and onion skinning

float rounded = d - r;       // expand by r (rounds/inflates shapes)
float onion   = abs(d) - r;  // hollow ring at distance r from surface

Rounding is achieved by subtracting a small value from the SDF — this inflates the shape and rounds its corners. Onion skinning takes the absolute value of a distance field to create concentric rings.

Groups

Collapsing complexity into a single node — and making that node run multiple times.

A group is a subgraph that collapses into a single node. The nodes inside are still there, still compiling, still contributing to the shader — they're just hidden behind a cleaner interface. Groups have wired input and output sockets, just like any other node.

Creating a group

  1. Select nodes Click nodes to select them, or drag a selection box. Selected nodes highlight.
  2. Group them Press ⌘G or right-click → "Group selected". A dialog asks you to name the exposed input and output ports.
  3. Result The selected nodes disappear into a single collapsed node. Double-click the group to expand and edit its internals.

By default, a group runs its subgraph exactly once per pixel — the same as if the nodes were inline.

The Iterations parameter

Every group has an Iterations parameter in its settings panel. Set it to anything greater than 1 and the group becomes an iterated group — its subgraph runs multiple times per pixel, with the output of each iteration feeding into the next as input.

This is Shader Studio's loop system. There are no separate loop nodes to wire together — just a group with Iterations > 1. See How Loops Work for the full picture.

Parameter overrides

Parameters of nodes inside a group can be promoted to the group's outer interface. When you expose an inner node's parameter, it appears as a socket on the group node itself — letting you override it from outside without opening the group.

How Loops Work

The most powerful tool in Shader Studio — and the one that most deserves a careful introduction.

There's something profound about repetition. A single circle is a circle. That same circle, evaluated six times with a slightly different coordinate each time and its outputs threaded through, becomes something approaching a fractal. The gap between those two statements — between one and six — is the loop.

Loops in Shader Studio are built on a single mechanism: iterated groups. A group with Iterations set to N runs its subgraph N times per pixel. That's the whole system. Everything else follows from it.

The core mental model

When you set a group's Iterations to 4, the compiler generates a GLSL for loop that runs your subgraph four times. On each pass:

  • The iteration number is available via a Loop Index node inside the group (outputs 0.0, 1.0, 2.0, 3.0)
  • Any node you mark as a carry node has its output variable persist across iterations — the result of one pass becomes the starting point for the next
  • All other nodes evaluate fresh each iteration, as if running independently

The key question is: what value are you threading through?

  • vec2 carry → transforming a UV coordinate through N iterations (domain folding, fractal UV)
  • vec3 carry → accumulating a color (summing contributions from N light sources, N ring glows)
  • float carry → accumulating a scalar (noise layers, distance field accumulation)

What actually gets compiled

Say you have a Fract node inside a group with Iterations=4, and you've marked Fract as a carry node. The compiler generates roughly this GLSL:

// Carry node's variable is declared OUTSIDE the loop, initialized from group input
vec2 fract_output = groupInput;

for (int i = 0; i < 4; i++) {
    float loopIndex = float(i);  // Loop Index node

    // The carry node runs using its own previous output as input
    fract_output = fract(fract_output * 1.5) - 0.5;

    // Non-carry nodes run normally, reading the carry value where wired
    float dist = length(fract_output) - 0.3;
}

// fract_output now holds the 4x-folded UV
// dist holds the distance from the final iteration

The key insight: a carry node's output variable persists between iterations because it's declared outside the loop. On each pass, the node writes back to its own variable. This is the carry mechanism — no separate node required, just a flag on any existing node.

The subgraph is not copied N times in source. The GPU runs a real loop, and carry variables thread state through it iteration by iteration.

Loop count is a compile-time constant. You cannot drive it from a Time node or any runtime value — Iterations is a slider on the group, not a socket. To change iteration count, adjust the group parameter and let the shader recompile.

Iterated Groups

How to build a group that loops.

Step by step: creating your first iterated group

  1. Build the single-pass version first Before making it loop, get the single-iteration version working on canvas. For example: UV → Fract (scale 1.5) → Circle SDF → Make Light → Output. Verify it looks right.
  2. Select the nodes you want to repeat Select Fract and Circle SDF (not the UV source, not Output). These are the nodes whose logic will run N times per pixel.
  3. Group them with ⌘G Give the group meaningful port names. You'll expose at minimum: an input UV (vec2) and an output color (vec3) or distance (float).
  4. Set Iterations in the group's parameter panel Select the group node and set Iterations to 4 (start here). The group now runs its subgraph 4 times per pixel.
  5. Double-click to enter the group, then mark the carry node Inside the group, right-click the node whose output you want to thread through iterations — for example, Fract. Toggle Carry Mode on. The node will visually indicate it's a carry node. That's it. No additional nodes required.

What marking a node as carry actually does

When Carry Mode is enabled on a node, the compiler declares that node's output variable outside the for loop, initialized from the group's input. Each iteration, the node computes normally — but because its output variable persists, the result feeds back as the input on the next pass.

For the Fract example:

Group Input (UV)
→ init
Fract [carry]
↺ feeds back each iteration

Iteration 0: Fract receives the original UV, outputs folded UV.
Iteration 1: Fract receives the folded UV from iteration 0, folds again.
Iteration 2: Fract receives the doubly-folded UV. And so on.

You can mark multiple nodes as carry inside the same group. A common pattern is to carry both a UV (for folding) and a color accumulator (for adding glow each pass) simultaneously — each carry variable is independent.

The Loop Index node

Inside an iterated group, a Loop Index node outputs the current iteration number as a float: 0.0 on the first pass, 1.0 on the second, and so on. This is what makes each iteration distinct — you can shift a palette phase, offset a position, scale a frequency, or use any expression that varies per pass.

// Classic use: offset palette phase per iteration
float phase = loopIndex * 0.4;  // each ring gets a different hue
vec3 color = palette(distance + phase, ...);

Without Loop Index, every iteration would read the same parameters and produce structurally identical results. Loop Index is the mechanism that makes iterative loops produce variety rather than pure repetition.

Start with Iterations = 3 or 4 while building. It's much easier to reason about what's happening with fewer passes. Once the pattern is correct, push to 6–8 for the final look.

Carry Mode

How a node's output threads through all iterations of a loop.

Carry mode is a toggle on individual nodes inside an iterated group. When enabled, that node's output variable persists across iterations instead of being recalculated fresh each pass. The output of iteration N becomes the input of iteration N+1 — automatically, by virtue of how the GLSL variable is declared.

How to enable it

Right-click any node inside an iterated group and toggle Carry Mode. The node will display a visual indicator that it's a carry node. That's the entire interface. There's no separate Loop Carry node to wire up — the carry is an attribute of the node itself.

What changes in the compiled shader

Normally, every node's output variable is declared inside the for loop, so it's fresh on each iteration. A carry-mode node's variable is instead declared before the loop begins, initialized from whatever feeds into the group's input port. Inside the loop, the node writes back to that same variable each pass.

// Non-carry node: variable declared inside the loop (fresh each iteration)
for (int i = 0; i < 4; i++) {
    vec2 regularNode_out = someFunction(input);  // recalculated from scratch
}

// Carry-mode node: variable declared outside the loop (persists)
vec2 carryNode_out = groupInput;  // initialized once before the loop
for (int i = 0; i < 4; i++) {
    carryNode_out = someFunction(carryNode_out);  // reads its own previous output
}

Multiple carry nodes

A single iterated group can have multiple nodes in carry mode simultaneously. Each has its own persistent variable, initialized from whatever is wired to it at the group level. A common pattern: mark a Fract node as carry (to thread a UV through folding passes) and also mark an Add Colors node as carry (to accumulate glow each iteration). The two carry variables are entirely independent.

What happens without any carry nodes

A group with Iterations > 1 but no carry nodes is valid. The subgraph runs N times, but every iteration starts from the same inputs and produces the same result — you effectively get N identical passes and the output of the last one. This is occasionally useful when you want N independent evaluations all driven by Loop Index (orbiting shapes, for example), but more often it's a mistake. If your loop looks like it's doing nothing, check that you've marked a carry node.

Carry mode and compound assignment

It helps to think of carry mode in terms of compound assignment operators. When a carry-mode node runs each iteration, it's conceptually doing this:

// UV folding carry (Fract node in carry mode):
uv = fract(uv * 1.5) - 0.5;  // = operator: replace with transformed value

// Color accumulation carry (Add Colors node in carry mode):
color += thisIterationGlow;    // += operator: add to running total

// Damping carry (Scale Color node in carry mode):
color *= 0.95;                 // *= operator: decay each pass

The operator that best describes the accumulation determines which node type to put in carry mode. If you're building up (+=), put an Add node in carry mode. If you're transforming (=), put the transform node itself in carry mode. If you're decaying (*=), a Scale node in carry mode does it.

The same operators — =, +=, -=, *=, /= — are available explicitly in the Expr Block node for single-node accumulation outside of loops too.

The most natural carry node is often the one that represents "the thing being transformed." For UV folding, that's your Fract or warp node. For color accumulation, that's your Add Colors node. Think about which node's output you want to "remember" between passes — that's your carry.

Loop Examples

Three concrete loop patterns, built from scratch.

Example 1: Fractal UV folding

The classic fractal rings effect. Each iteration folds the UV space, evaluates a ring SDF, and accumulates a color glow. The UV gets denser and more self-similar each pass, producing detail at multiple scales simultaneously.

Carry nodes: Fract (vec2, threads the UV through) + Add Colors (vec3, accumulates glow each pass)

Iterations: 5–8

Inside the group:

  1. Add a Fract node — set scale to 1.5. Right-click → enable Carry Mode. It will receive the initial UV on the first pass, then its own folded output on each subsequent pass.
  2. Wire Fract's output to a Circle SDF.
  3. Wire Loop Index (scaled by 0.4) into a Palette node's T input, offset by the distance — each iteration gets a different hue phase.
  4. Wire distance into a Make Light node → glow intensity.
  5. Scale the palette color by the glow.
  6. Add an Add Colors node. Right-click → enable Carry Mode. It starts at vec3(0.0) and accumulates each iteration's color contribution.
  7. Wire the scaled color into Add Colors' second input. Its output is the running total.
// Compiled GLSL — Fract and Add Colors are both carry-mode nodes:
vec2 fract_out = inputUV;          // Fract carry, initialized from group input
vec3 addcol_out = vec3(0.0);       // Add Colors carry, starts at black

for (int i = 0; i < 6; i++) {
    float idx = float(i);
    fract_out = fract(fract_out * 1.5) - 0.5;   // folds and persists
    float d = length(fract_out);
    float ring = sin(d * 8.0 + u_time) / 8.0;
    float glow = 0.01 / abs(ring);
    vec3 col = palette(d + idx * 0.4 + u_time * 0.4, ...);
    addcol_out += col * glow;                     // accumulates and persists
}
// addcol_out is the final color output from the group

Example 2: Layered noise (manual FBM)

Build Fractal Brownian Motion manually — each iteration samples the same noise at a higher frequency and lower amplitude, layering detail from coarse to fine.

Carry node: An Add (float) node that accumulates the noise sum

Iterations: 5

Inside the group:

  1. Add an Expr node that computes pow(2.0, loopIndex) — this is the frequency for this pass.
  2. Multiply UV by frequency → scaled UV for noise sampling.
  3. Wire scaled UV into a Value Noise node → raw noise float.
  4. Compute amplitude: pow(0.5, loopIndex). Multiply by raw noise → weighted contribution.
  5. Add a float Add node. Right-click → enable Carry Mode. It accumulates the weighted noise each iteration, starting from 0.0.
float noiseSum = 0.0;   // Add node in carry mode, initialized to 0
for (int i = 0; i < 5; i++) {
    float freq = pow(2.0, float(i));
    float amp  = pow(0.5, float(i));
    noiseSum += amp * valueNoise(uv * freq);
}

Example 3: Orbiting shapes

Place N shapes arranged in a ring, each given a distinct position via Loop Index. Accumulate all their glows into a single color output.

Carry node: Add Colors (vec3) that accumulates each shape's glow contribution

Iterations: 6

Inside the group:

  1. Loop Index → compute angle per shape: loopIndex * 6.283 / 6.0 + time * 0.5
  2. From angle: vec2(cos(angle), sin(angle)) * radius → position of this shape
  3. Subtract from UV → local coordinate centered on this shape's position
  4. Circle SDF → distance to this shape
  5. Make Light → glow
  6. Palette with loopIndex / 6.0 + time * 0.2 → this shape's hue
  7. Scale Color by glow → this shape's color contribution
  8. Add Colors node with Carry Mode enabled → running total of all shapes
vec3 colorSum = vec3(0.0);   // Add Colors in carry mode
for (int i = 0; i < 6; i++) {
    float angle = float(i) * 6.283 / 6.0 + u_time * 0.5;
    vec2 pos = vec2(cos(angle), sin(angle)) * 0.5;
    float d   = length(uv - pos) - 0.08;
    float g   = 0.01 / abs(d);
    vec3  col = palette(float(i) / 6.0 + u_time * 0.2, ...);
    colorSum += col * g;
}
Start loops with 3–4 iterations while building. They're easier to reason about. Once the pattern is right, push to 6–8 for the final look.

Noise

Controlled randomness — the texture of the natural world.

Pure math produces forms that are too perfect. Real clouds don't have uniform density. Real fire doesn't burn symmetrically. Noise is the discipline of making math imperfect in beautiful, controllable ways.

FBM — Fractal Brownian Motion → float

FBM layers multiple octaves of value noise at increasing frequencies and decreasing amplitudes. Each octave adds finer detail at lower intensity, producing the characteristic "natural" texture seen in clouds, terrain, and organic surfaces.

// FBM at its core:
float fbm(vec2 p, int octaves) {
    float value = 0.0;
    float amp   = 0.5;
    float freq  = 1.0;
    for (int i = 0; i < octaves; i++) {
        value += amp * valueNoise(p * freq);
        amp   *= gain;       // typically 0.5 — each octave is quieter
        freq  *= lacunarity; // typically 2.0 — each octave is finer
    }
    return value;
}

Parameters:

  • Octaves — how many layers. More = more detail but slower. 4–6 is the sweet spot.
  • Lacunarity — frequency multiplier per octave. 2.0 doubles frequency each time (standard).
  • Gain — amplitude multiplier per octave. 0.5 halves amplitude each time (standard).
  • Scale — overall spatial scale of the noise.
  • Anim Speed — drives the noise through time by sliding the UV in a diagonal direction.

FBM outputs a float in approximately [0, 1]. Wire it into palette T for colored noise, use it as a displacement amount for the Displace node, or subtract 0.5 and use it as a signed warp offset.

Voronoi / Worley noise → float

Voronoi noise divides space into irregular cells, one per randomly placed seed point, and returns the distance to the nearest cell center. The result looks organic — cells, cracks, cobblestones.

  • Jitter — how randomly the cell centers are placed. 0 = regular grid, 1 = fully random.
  • Scale — number of cells across the canvas.

The raw distance output (f1) gives a smooth gradient inside each cell. For crack/vein patterns, you want the difference between the two nearest cells (f2 - f1) — this is brightest at cell edges. Use an Expr node: f2 - f1 where you compute the second nearest distance.

// Crack / vein pattern from Voronoi:
float veins = voronoi_f2(uv) - voronoi_f1(uv);
float brightness = 0.02 / max(veins, 0.001);

Domain Warp → vec2

Domain warping feeds the output of one noise evaluation back as the input UV for the next. The result is dramatically more complex and organic than either noise alone — the characteristic look of lava, storm systems, and wet paint.

// Two levels of domain warp:
vec2 q = vec2(fbm(p),             fbm(p + vec2(5.2, 1.3)));
vec2 r = vec2(fbm(p + 4.0*q),     fbm(p + 4.0*q + vec2(1.7, 9.2)));
float f = fbm(p + 4.0*r);

Apply domain warp 2–3 levels deep for maximum complexity. Each level wraps around the previous, producing recursively distorted forms.

Wire Time at a slow speed (0.1–0.2) into the noise nodes inside a Domain Warp setup. The result animates like a living fluid — fast time makes it look like boiling, slow time looks like glacial drift.

Color

From raw floats to rich, animated palettes.

IQ Cosine Palette

The Palette node implements Inigo Quilez's cosine palette formula — perhaps the most elegant color tool in shader programming:

vec3 palette(float t, vec3 a, vec3 b, vec3 c, vec3 d) {
    return a + b * cos(6.28318 * (c * t + d));
}

Feed a float into t and the formula cycles through a rich color space. The four vec3 parameters control the character of the palette:

  • Offset (a) — the baseline brightness of each channel. vec3(0.5) centers everything.
  • Amplitude (b) — how much each channel varies. vec3(0.5) fills the full range.
  • Frequency (c) — how many times each channel cycles per unit of t. vec3(1.0) cycles once.
  • Phase (d) — the phase offset of each channel. vec3(0.0, 0.33, 0.67) spreads R/G/B 120° apart for a rainbow.

Wire Time into t for cycling animation. Wire an SDF distance into t for distance-based coloring. Wire Loop Index into t (offset per iteration) for per-ring color variation in loops.

Palette Preset

10 curated palette configurations ready to use: IQ Blue-Teal, IQ Rainbow, IQ Warm, IQ Lemon, Sunset, Fire, Forest, Purple Haze, Deep Purple, Psychedelic. Use this for quick results; switch to the manual Palette node when you want to dial in specific colors.

Gradient → vec3

Blends between two colors across UV space. Five modes: Linear X, Linear Y, Radial (by distance from center), Angular (by atan2 angle), Diagonal. The T Offset parameter shifts the blend point. Wire a float into the Color A and Color B inputs to animate the colors themselves.

Tone Map → vec3

HDR colors (values above 1.0) need to be mapped back into displayable range [0,1]. Without tone mapping, bright areas clip to white. The Tone Map node applies a curve that preserves highlight detail:

  • ACES — filmic S-curve. Slightly desaturates highlights and deepens shadows. Most cinematic result. Use this by default.
  • Reinhard — simple x / (1 + x). Preserves color hue, gentle rolloff.
  • Filmic / Hable — stronger contrast in the mids, good for stylized looks.
  • Exposure — pure brightness multiplier, no curve. Useful for debugging actual values.
Place a Tone Map node as the last step before Output whenever your shader uses Make Light, glow formulas, or accumulation loops. Without it, hot pixels blow out to white and you lose the color detail in bright areas.

HSV ↔ RGB conversion

The RGB to HSV and HSV to RGB nodes convert between color spaces. Working in HSV is useful when you want to shift hue independently, animate saturation, or create color wheels. The H component cycles: 0 and 1 are both red.

Mix, Clamp, Posterize, Invert

  • Mix — linear interpolation between two vec3 colors. Wire a float into T (0 = all A, 1 = all B).
  • Clamp — restricts a value to [min, max]. Essential before Output if your shader produces out-of-range values without tone mapping.
  • Posterize — quantizes color into N discrete bands. Creates a cel-shaded / retro look.
  • Invert — 1.0 - color. Flips all channels.
  • Gamma — applies a power curve. Gamma < 1 brightens midtones, Gamma > 1 darkens them.

Math Nodes

The arithmetic layer that connects everything else.

Arithmetic

  • Add, Subtract, Multiply, Divide — basic float operations. Each has a B slider as fallback when the B socket is unwired.
  • Add Colors — adds two vec3 values. The accumulation workhorse for multi-light setups and loop color building.
  • Scale Color — multiplies a vec3 by a float. Use this to apply a glow intensity to a palette color.

Trigonometry

  • Sin, Cos — outputs amp * sin(input * freq). Both the amplitude and frequency can be overridden by sockets. Wire Time into input for oscillation.
  • All trig functions work in radians. One full rotation = 2π ≈ 6.2832.

Shaping functions

  • Step — hard threshold: step(edge, x) = 0 if x < edge, 1 if x ≥ edge. Creates a sharp binary boundary.
  • Smoothstep — hermite interpolation between two edges. Produces a smooth 0→1 transition. The most useful shaping function in the library.
  • Mix — linear interpolation: mix(a, b, t). Blends any two values.
  • Clamp — restricts a value to a range.
  • Remap — maps a value from one range to another: remap(x, inMin, inMax, outMin, outMax).

Component operations

  • Abs — absolute value. Mirrors negative values to positive. Core of onion-skinning: abs(d) - r.
  • Floor, Ceil, Fract — decompose a float into integer and fractional parts.
  • Mod — modulo. The basis of tiling and repetition.
  • Sign — returns -1.0, 0.0, or 1.0. Useful for extracting directionality from a signed distance.
  • Pow — base raised to exponent. Use for gamma curves and sharp shaping functions.
  • Sqrt, Exp, Log — standard math functions.

Vector operations

  • Length — distance from a vec2 to the origin. The Circle SDF formula is just length(uv) - radius.
  • Normalize — scales a vector to length 1.
  • Dot — dot product of two vectors. See Vectors & Matrices.
  • Split — separates a vec2/vec3 into its individual float components.
  • Join — combines two or three floats into a vec2 or vec3.
  • Swizzle — reorder or duplicate components: turn a vec3 (r,g,b) into vec3 (b,g,r).

Animation & LFOs

Time as raw material.

Animation in shaders is nothing more than making output depend on time. Wire the Time node into anything — a palette's T input, a Rotate 2D's angle, a noise's offset — and that thing becomes animated. Every parameter is a potential dimension of motion.

LFOs (Low Frequency Oscillators) are nodes that transform raw time into specific oscillation patterns. They take a Time input and output a value that moves predictably between a min and max at a given frequency.

Sine LFO → float

Smooth sinusoidal oscillation. The most natural-feeling LFO — it eases in and out at each extreme, like breathing. Use for scale pulsing, opacity breathing, gentle color shifts. Frequency is in Hz (cycles per second).

Square LFO → float

Instant switch between min and max. No transition. Use for strobing, binary state changes, rhythm-locked flashing. The duty cycle parameter (0–1) controls how long it spends in the "high" state.

Sawtooth LFO → float

Linear ramp from min to max, then instant reset. Like a countdown that resets. Use for looping motion, scrolling textures, anything that needs to sweep and restart. Phase controls where in the cycle it starts.

Triangle LFO → float

Symmetric ramp up then ramp down. Like Sawtooth but without the sharp reset — it reverses at both ends. Slightly more natural than Sawtooth for back-and-forth motion.

BPM Sync

Takes a BPM value and outputs a normalized phase [0, 1] that completes one cycle per beat. Wire this into any LFO's phase input or use it directly as a palette T driver to lock your visuals to a music tempo. At 120 BPM, one cycle completes every 0.5 seconds.

Layer multiple LFOs at different frequencies for complex motion that never quite repeats. For example: a slow Sine at 0.1 Hz driving scale, plus a faster Sine at 0.37 Hz driving rotation, produces motion that takes over 10 seconds to approximately cycle.

Effects

Post-processing that happens after the main computation.

Make Light → float

The foundational glow formula: brightness / abs(distance). Feed any SDF distance into Distance, adjust Brightness to control how intense the glow is. The output is a scalar intensity — multiply it against a color to get a tinted glow. Multiple Make Light nodes can be added together for multi-light scenes.

Tone Map → vec3

Maps HDR values (above 1.0) into displayable range. See Color for full details on the available modes. Always place this before Output when using accumulation loops or glow.

Chromatic Aberration

Offsets the R, G, and B channels by slightly different amounts, simulating the color fringing seen in cheap camera lenses. The offset increases radially from the center, strongest at screen edges. Strength parameter controls offset magnitude. Wire a subtle amount (0.005–0.02) for a subtle analog feel.

Film Grain

Adds high-frequency random noise to the output. Animates each frame so it looks like physical film grain rather than a static texture. Intensity parameter. Best applied as the very last step before Output, after tone mapping.

Gravitational Lens

UV distortion that bends rays around a center point, imitating the lensing effect of a massive object. The distortion is strongest close to the lens center and falls off with distance. Strength, Center X, Center Y, and Falloff parameters.

Fractals

Infinite complexity from finite rules.

Fractal nodes run iterative algorithms that produce self-similar patterns at all scales of zoom. They're the computational cousins of the iterated group — both use repetition to build complexity — but fractal nodes are pre-packaged for specific mathematical systems.

Fractal Loop (Fractal Rings)

The compound "Fractal Rings" node packages the classic IQ fractal ring pattern — UV folding + ring glow accumulation — in a single pre-built node. Connect UV and Time, adjust iterations, scale, and ring frequency. For full control over the loop behavior, build it manually with an iterated group instead.

Mandelbrot → float

Iterates the formula z = z² + c where c is the pixel coordinate and z starts at 0. Returns the normalized escape count — how quickly the sequence diverges. Wire into a Palette node for classic Mandelbrot coloring.

  • Max Iterations — higher = more detail at cost of performance. 64–128 is typical.
  • Zoom — scale into the fractal. Animate with Time for a zoom sequence.
  • Center X/Y — the point you're zooming into.

Julia Set → float

Like Mandelbrot but with a fixed complex parameter c rather than using the pixel coordinate. Different c values produce radically different shapes — from connected islands to fractal dust. The C X and C Y parameters define the complex constant. Animate them slowly with Time for morphing shapes.

Burning Ship → float

Mandelbrot variant: z = (|Re(z)| + i|Im(z)|)² + c. The absolute values before squaring create a completely different fractal geometry — resembling a burning ship when oriented correctly.

Apollonian Gasket → float

Iterated circle inversions. Each iteration inverts the coordinate through a set of circles, producing a fractal arrangement of tangent circles packed at all scales. Feed output into Palette for classic gasket coloring.

Menger Sponge (3D) → float

3D SDF fractal. Produced by iterating box subtraction at decreasing scales. Use inside a 3D scene via the Scene Group. The iteration count controls how many levels of sponge holes appear — 3–4 is usually sufficient.

All fractal nodes output a float in [0, 1] representing the iteration ratio or escape time. This float is designed to be fed into Palette — that's the standard pattern for getting color from a fractal computation.

Ray Marching

3D without triangles — the complete pipeline.

Traditional 3D rendering works by rasterizing triangles — projecting geometry vertices onto the screen. Ray marching works differently: for each pixel, it shoots a ray from the camera into the scene and asks, over and over, "am I close to any surface yet?"

The answer at each step comes from an SDF — the same signed distance function you've been using in 2D, extended into 3D. The SDF tells the ray how far it can step safely without missing a surface. When that distance gets very small, the ray has hit something.

The march loop

vec3 rayOrigin = cameraPosition;
vec3 rayDir    = normalize(cameraMatrix * vec3(screenUV, focalLength));

float totalDist = 0.0;
for (int i = 0; i < MAX_STEPS; i++) {
    vec3 p = rayOrigin + rayDir * totalDist;
    float d = sceneSDF(p);           // distance to nearest surface
    if (d < SURFACE_THRESHOLD) break; // we've hit something
    if (totalDist > MAX_DIST)   break; // ray escaped the scene
    totalDist += d;                  // safe to step this far
}

The insight is that stepping by exactly d at each iteration is always safe — by definition of the SDF, there is no surface within distance d of the current point. This is called sphere tracing, and it's why SDF-based ray marching converges quickly.

Surface normals

A surface normal at a point is computed numerically — by sampling the SDF at tiny offsets and computing the gradient:

vec3 calcNormal(vec3 p) {
    float e = 0.001;
    return normalize(vec3(
        sceneSDF(p + vec3(e,0,0)) - sceneSDF(p - vec3(e,0,0)),
        sceneSDF(p + vec3(0,e,0)) - sceneSDF(p - vec3(0,e,0)),
        sceneSDF(p + vec3(0,0,e)) - sceneSDF(p - vec3(0,0,e))
    ));
}

The Raymarch 3D node computes this automatically. When you receive the Normal output, it's already normalized and ready for lighting calculations.

3D SDFs

The geometry vocabulary of ray marching.

Every shape in a ray marched scene is defined by a signed distance function in 3D space. The same principles as 2D SDFs apply — negative inside, positive outside, zero on the surface — but now operating on vec3 positions.

Primitive nodes

  • Sphere SDF 3D — length(p) - r. The fundamental 3D primitive.
  • Box SDF 3D — axis-aligned box with independent X/Y/Z half-sizes.
  • Torus SDF 3D — a donut. Major radius R (ring size), tube radius r (thickness).
  • Cylinder SDF 3D — capped cylinder with radius and half-height.

3D Lighting

How surfaces become visible.

A surface hit by a ray has a position and a normal. Lighting turns those two pieces of information into a color. The Raymarch 3D node handles this internally, but understanding the components lets you customize the output using the Normal and Depth sockets.

Diffuse lighting

The most fundamental lighting model: how much does the surface face the light?

vec3 normal   = calcNormal(hitPoint);
vec3 lightDir = normalize(vec3(1.0, 1.5, 0.5)); // light direction
float diffuse = max(dot(normal, lightDir), 0.0);
vec3 color = surfaceColor * diffuse;

Specular (Phong)

The bright highlight that appears when the surface is at the right angle relative to both the light and the viewer:

vec3 viewDir  = -rayDir; // camera is looking from here
vec3 reflect  = reflect(-lightDir, normal);
float spec    = pow(max(dot(viewDir, reflect), 0.0), shininess);
vec3 color   += specularColor * spec;

Ambient occlusion

AO darkens crevices and corners where light can't easily reach. It's estimated by sampling the SDF at small offsets along the normal — nearby surfaces reduce the ambient contribution. The Raymarch 3D node's Occlusion output carries this value for use in custom shading.

Soft shadows

Ray march a second ray from the surface toward the light. If it hits something before reaching the light, the surface is in shadow. Soft shadows penumbra by tracking the closest approach distance during the shadow ray march.

Fog

Blend the surface color with a sky/background color based on depth: mix(surfaceColor, fogColor, fogFactor) where fogFactor = 1 - exp(-depth * density). The Fog Mask output from Raymarch 3D gives you this factor ready to use.

Volumetrics

Rendering the space between surfaces — fog, clouds, god rays.

Standard ray marching finds the first surface a ray hits. Volumetric rendering accumulates contributions along the entire ray path — sampling the density of a medium at every step. This is how clouds, fog, fire, and god rays are rendered.

The volumetric march

float totalDensity = 0.0;
vec3  totalColor   = vec3(0.0);
float stepSize     = 0.05;

for (int i = 0; i < 64; i++) {
    vec3 p = rayOrigin + rayDir * float(i) * stepSize;
    float density = cloudDensity(p);       // sample the volume
    totalDensity += density * stepSize;
    totalColor   += lightColor * density * stepSize * exp(-totalDensity);
}
// totalColor is the accumulated in-scattered light

The exp(-totalDensity) term is Beer's Law — it models how light is absorbed as it travels through matter. Each step contributes less color as the accumulated density grows.

Cloud density function

The shape of a volumetric cloud is defined by a scalar density function — how thick the medium is at each 3D point. A typical cloud density uses turbulent noise modulated by height:

float cloudDensity(vec3 p, float coverage, float puffiness, float scale) {
    float base = -p.y * 0.5 + coverage;   // height falloff
    float turb = turbulence(p * scale, 4); // layered noise
    float d = base - turb * puffiness;
    return max(d, 0.0);                   // can't have negative density
}

The Raymarch 3D node in volumetric mode

The Raymarch 3D node includes a "clouds" scene type that runs volumetric accumulation. Select it from the Scene dropdown. Parameters include Coverage (how thick the cloud layer is), Puffiness (how much turbulence shapes the cloud), and Noise Scale (feature size). The output color includes in-scattered light.

God rays / light shafts

Volumetric light shafts are produced by ray marching toward a light source and accumulating density that hasn't been occluded. At each step, cast a shadow ray to the light — if it's not blocked, add to the accumulation. The characteristic "beams through clouds" look comes from varying density along the primary ray creating alternating lit and shadowed regions.

Building volumetrics with iterated groups

You can approximate volumetric accumulation using an iterated group. Each iteration represents one march step along the ray. Use a float carry to accumulate density and a vec3 carry to accumulate color. Connect Loop Index to compute the step position along the ray.

// Per iteration inside the group:
float t = loopIndex * stepSize;             // position along ray
vec3  p = rayOrigin + rayDir * t;
float density = noiseAt(p);
colorCarry += lightColor * density * stepSize;  // accumulate
densityCarry += density * stepSize;

Function Builder

Build reusable GLSL functions visually, then call them by name from anywhere.

The Function Builder is a separate canvas — accessible from the top navigation — where you define named GLSL functions. A function you build here shows up in the node palette as a Custom Function node. Wire it in like any other node.

The value here is reuse. If you have a 12-node computation you use in three different places, put it in the Function Builder once, then call it three times. Change it once, all three callers update.

Building a function

  1. Open Function Builder Click the Function Builder tab in the top navigation.
  2. Name your function Give it a descriptive name — this becomes the node's label and the GLSL function name.
  3. Add inputs Click "Add Input" to declare typed parameters (float, vec2, vec3, vec4). Each becomes an input socket on the resulting node and an argument in the generated GLSL function signature.
  4. Build the computation Use any nodes in the Function Builder canvas. The inputs are available as source nodes. Connect everything to the Function Output node.
  5. Set output type On the Function Output node, select the return type of your function.
  6. Save Click Save. The function appears in the node palette under Custom Functions.

Using your function in the main graph

Find it in the palette (search by name) and wire it just like any built-in node. The input sockets correspond to the arguments you declared, and the output matches the return type you set.

Editing an existing function

Open Function Builder, select the function from the list on the left, make changes, and save again. All instances of that function in the main graph automatically reflect the update on the next recompile.

Practical use cases

  • Custom SDF — build an unusual 2D shape from primitives, save it as a function, reuse it tiled with opRepeat.
  • Palette wrapper — a function that takes a float and returns a vec3 using your specific palette parameters, so you can reuse the same color scheme in multiple places without re-specifying all 12 palette parameters.
  • Noise composite — a function that layers FBM + Voronoi + domain warp in a specific way, accepts UV and Time as inputs, outputs a float. Call it anywhere you want that specific texture.

GLSL / Expression Editor

When nodes run out — write it yourself.

The Expr node is a node with a built-in GLSL code editor. You declare typed input ports, write a GLSL expression or body, and the result appears as an output socket. The node is fully wireable — it fits into any graph just like a built-in node.

Adding an Expr node

Open the node palette and search "Expr" or "Expression". Double-click the node to open its editor. You'll see:

  • An Inputs section — declare named inputs with types
  • A Body area — optional intermediate computations
  • A Result field — the final expression to return
  • An Output Type selector — float, vec2, vec3, or vec4

Writing the expression

The inputs you declare appear as variables in scope. Just use their names:

// Inputs declared: d (float), t (float)
// Output type: float
// Result:
0.01 / abs(sin(d * 8.0 + t) / 8.0)
// Inputs: uv (vec2), t (float)
// Output type: vec2
// Result:
vec2(sin(t) * 0.5, cos(t * 1.3) * 0.4) + uv * 0.8
// Inputs: uv (vec2)
// Output type: float
// Body:
float angle = atan(uv.y, uv.x);
float r = length(uv);
// Result:
sin(angle * 6.0) * r

Assignment operators in the Body

Each line in the Body section has three parts: a left-hand side variable, an operator, and a right-hand side expression. The operator dropdown offers five choices:

OperatorMeaningEquivalent to
=Assignx = expr — replace x with the result
+=Add and assignx = x + expr — add expr to x and store back
-=Subtract and assignx = x - expr — subtract expr from x and store back
*=Multiply and assignx = x * expr — scale x by expr and store back
/=Divide and assignx = x / expr — divide x by expr and store back

The compound operators (+=, -=, *=, /=) are only meaningful when the left-hand side is a variable you declared earlier in the Body — they read its current value, transform it, and write back. Using += on an undeclared variable is a compiler error.

// Example: accumulate three offsets into a position
// Inputs: uv (vec2), t (float)
// Body:
vec2 p = uv;          // p = uv        (declare p)
p += vec2(0.1, 0.0);  // p += ...      (shift right)
p *= 1.5;             // p *= 1.5      (zoom in)
p -= vec2(0.0, sin(t) * 0.2);  // p -= ... (oscillate vertically)
// Result:
p

This is particularly useful for building up coordinate transforms step by step — each line applies a transformation to the running variable without needing to write the full accumulated expression in one shot.

Compound operators also apply to individual components. p.x += 0.5 shifts only the x component of a vec2 or vec3. color.rgb *= 0.8 darkens a vec4's color channels without touching alpha. This works anywhere GLSL swizzle assignment is valid.

Available built-ins

All standard GLSL functions are available: sin, cos, tan, atan, sqrt, pow, exp, log, abs, sign, floor, ceil, fract, mod, min, max, clamp, mix, step, smoothstep, length, normalize, dot, cross, reflect.

Shader Studio also injects its built-in helpers: palette(), sdBox(), sdSegment(), sdEllipse(), opRepeat(), opRepeatPolar(), rotate(), smin(), valueNoise(), fbm().

Uniforms like u_time and u_resolution are also available — but for clean graph design, prefer wiring Time and UV in as inputs rather than referencing uniforms directly.

Common errors

  • Type mismatch — float + vec2 doesn't compile. Be explicit: vec2(f) + v or use swizzling.
  • Integer literals — write 2.0 not 2. 6.0 / 2 is a type error in strict GLSL.
  • Missing semicolons — every statement in the Body section needs a ;. The Result field is an expression (no semicolon).
  • Undeclared variables — if you use a variable in the Result that you didn't declare in Body or Inputs, it won't compile.
The GLSL panel (⌘\) shows how your Expr node compiles. If you're getting unexpected results, open it and find your node's output variable — see exactly what expression is being evaluated.

Glass & Refraction

Physical glass, chromatic dispersion, and spectral color splitting — as nodes.

Three new lighting nodes handle the full repertoire of transparent material effects: Glass 3D for complete physically-based glass, Spectral Dispersion for six-band rainbow splitting, and Blinn-Phong as a standalone lighting scalar.

Glass 3D

A self-contained glass material. Give it a ray direction, a surface normal, and a hit flag — it handles Fresnel blending, multi-sample chromatic dispersion, environment sampling, and optional Blinn-Phong specular/diffuse in a single node.

Auto-wiring on add

When you add a Glass 3D node from the palette, Shader Studio scans the graph for an existing March Camera and March Loop Group. If found, it automatically wires rayDir ← cam.rd, normal ← mlg.normal, and hit ← mlg.hit. If no march setup exists, it spawns the full set — camera, scene group, loop group, and glass node — all pre-wired.

Typical wiring

UV→ March Camera→ March Loop Group→ Glass 3D→ Tone Map → Output

Leave lightDir unconnected for pure refraction, or wire a Vec3 Const like (0.6, 1.0, 0.4) to add highlights.

The environment

Glass 3D uses a procedural dot-field environment — a two-layer grid of white dots on a tinted sky, mapped equirectangularly from the ray direction. This means refraction and reflection are visually apparent even in a completely empty scene: the dots appear distorted through the glass, and the chromatic channels shift by slightly different angles.

Spectral Dispersion

A more physically accurate dispersion model using six spectral bands: Red, Yellow, Green, Cyan, Blue, Violet (rygcbv). Each band refracts at its own IOR, and the six results are recombined via Fourier interpolation into an RGB output. This produces richer rainbow splitting than the 3-channel model in Glass 3D.

Inputs & Parameters

Same rayDir / normal / hit / bgColor inputs as Glass 3D. Six individual IOR sliders — iorR through iorV — let you dial in exactly how much each spectral band bends. Typical glass spans roughly 1.505 (red) to 1.538 (violet).

Use Spectral Dispersion when you want rainbow-edge prismatic effects. Use Glass 3D when you want the full package — Fresnel, lighting, and dispersion — in one node.

Halftone

Print-process aesthetics — dots, rings, and CMYK screens — built from composable nodes.

Halftone effects simulate the ink-dot printing process: a continuous-tone image is broken into a grid of dots whose size encodes local brightness. In Shader Studio these are built from low-level composable nodes — so you can mix patterns, drive dot size from any signal, and layer screens at different angles.

Core nodes

Grid UV

Divides UV space into a repeating grid of cells. Outputs the UV position within the current cell (cellUV, 0–1) and the integer cell index (cellIndex). Wire cellUV to any SDF or mask to tile it across the canvas.

  • Scale — number of cells across the canvas.
  • Stagger — offset every other row by half a cell (0 = square grid, 1 = hex-offset).

Dot Mask

Draws a filled circle in each grid cell. The radius input controls per-cell dot size — wire a noise-driven Luma Radius value here for luminance-controlled halftoning.

  • cellUV — wire from Grid UV.
  • radius — float per-cell dot radius, range 0–0.5.
  • softness — anti-alias the dot edge.

Luma Radius

Converts a vec3 color to luminance, then remaps it to a radius value. This is the bridge between an image signal and halftone dot size.

  • baseRadius — radius at 50% grey.
  • minScale / maxScale — how much the dark and bright extremes compress or expand dot radius.

Ring SDF

Outputs the signed distance to a ring (annulus) in cell UV space. Combine with SDF Mask for hollow-circle halftones.

SDF Mask

Thresholds a signed distance field into a 0/1 mask with a configurable soft edge. Use after Ring SDF, or after any custom 2D SDF.

CMYK Halftone

A complete 4-channel halftone in one node. Takes a vec3 color and UV, decomposes to CMYK, screens each channel at a different angle (classic print rosette), and composites onto a paper color. Parameters include grid size, dot radius, softness, paper color, and screen angles for each channel.

Composing a halftone

UV + Time→ FBM→ Luma Radius→ Dot Mask→ Multiply→ Output

Wire Grid UV into Dot Mask's cellUV. Wire FBM → Float to Vec3 → Luma Radius → Dot Mask's radius. Multiply the mask by a Palette node for colored dots.

For animated halftones, drive the FBM with Time so the noise — and therefore the dot sizes — slowly shift over time.

Example graphs

  • Halftone Noise — FBM luminance drives dot radius; palette colors the dots.
  • CMYK Halftone — palette noise through the full 4-channel CMYK screen node.
  • Ring Halftone — ring SDFs + SDF Mask replace solid dots for a hollow-circle look.

GPU Particle Pipeline

Thousands of points, fully on the GPU, composed from a chain of nodes.

The GPU particle system renders large point clouds — galaxies, nebulae, breathing spheres, ring vortices — without touching the fragment shader at all. Particles are simulated and drawn entirely in WebGL, driven by a linear pipeline of particle nodes that each modify the point cloud in a composable way.

The pipeline model

A particle graph is a chain. Each node receives a particle stream, transforms or colors it, and passes it on. The chain always starts with Particle Init and ends with Particle Render.

pInit→ pRotate→ pWave→ pColorDist→ pSize→ pRender
You also need a standard Output node in your graph for the background. Wire a Vec3 Const (e.g. black) into Output so the canvas has a clear color for the particles to render over.

Glass Scene

Glass composited over a real 3D scene — with per-channel chromatic aberration that bends geometry, not just color.

Glass Scene is a self-contained rendering node that marches two separate scene SDFs: one for the glass geometry (foreground) and one for what's seen through and around it (background). Unlike glass3d, which uses a procedural dot environment, Glass Scene renders an actual scene behind the glass.

Wiring

March Camera→ ro / rd Scene Group A (glass geo)→ foreground Scene Group B (background)→ background Glass Scene→ Output

The background input is optional. When nothing is connected, a blue-to-dark sky gradient is used — chromatic aberration still applies to the sky sampling.

Parameters

ParamDefaultDescription

Chromatic aberration

When Dispersion > 0, three independent background marches fire — one per color channel — each with a slightly offset refracted ray direction. The R channel bends one way, B bends the other, and G stays near center. This means background geometry edges appear split by color, not just the sky. The effect scales from center to rim: the center stays clean while edges show the most splitting.

At Dispersion = 0, all three channels share one refracted ray. Zero aberration, and three fewer march loops per pixel.

GI Lit March Group

A ray march container with real-time global illumination: AO, soft shadows, sky dome IBL, one diffuse GI bounce, and one specular reflection — all in a single shader pass.

The GI Lit March Group works exactly like a March Loop Group — same enterable subgraph, same SceneGroup-inside workflow — but replaces the basic Lambert shading with a multi-pass lighting approximation that produces results that look like path tracing at interactive frame rates.

Quick start

March Camera→ ro / rd GI Lit March Group→ color Tone Map→ Output

Double-click the node to enter its body. Inside, place your SDF primitives connected through a Scene Group, then wire marchPos → [SDF / warps] → Scene Group → Group Output. The GI lighting is automatic — you only build the geometry.

Performance budget

Each pixel runs: 1 primary march + 1 AO march + 1 shadow march + 1 GI bounce march + 1 specular march = 5× the march cost of a plain MLG at equivalent step counts. Keep step counts lean and use Jitter to hide noise instead of raising steps.

Typical budget: Max Steps 64–96 · Shadow Steps 16–24 · GI Steps 12–16 · Spec Steps 16–24. Enable Jitter (0.5–1.0) to break up banding at lower step counts.

Tan node

Tan (amp * tan(input * freq)) was added to Math › Trig alongside Sin and Cos. It has the same signature — input, freq, amp — and supports float, vec2, and vec3 inputs.

The inline viz shows S-curves with dashed asymptote guides at ±π/2. The waveform pen lifts at each discontinuity so no false vertical lines are drawn at the asymptotes.

Tan grows much faster than Sin near ±π/2. Keep freq below 0.4 for smooth results; values above 0.45 will clip to the ±∞ asymptote within the visible UV range.
